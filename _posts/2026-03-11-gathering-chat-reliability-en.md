## Problem

Guaranteeing **at-least-once** delivery with the Outbox Pattern inevitably brings two problems:

1. **Message ordering** — Kafka only guarantees order within a partition. If messages from the same chat room land in different partitions, order can be scrambled.
2. **Duplicate consumption** — At-least-once means the same message may arrive more than once. Duplicate ChatMessage saves must be prevented.

## Action

### chatRoomId as Partition Key — Per-Room Order Guarantee

```java
public void publishEvent(Event<EventPayload> event) {
    EventPayload payload = event.getPayload();
    String topic = event.getType().getTopic();
    String partitionKey = String.valueOf(payload.getPartitionKey());

    kafkaTemplate.send(topic, partitionKey, DataSerializer.serialize(payload));
}
```

Messages with the same `chatRoomId` always go to the same partition, **guaranteeing per-room message ordering**. Messages from rooms A and B are processed independently in different partitions, maintaining parallelism.

### Redis Idempotency — Preventing Duplicate Consumption

```java
@Service
public class ChatService {

    public void handle(Event<ChatMessagePayload> event) {
        ChatMessagePayload payload = event.getPayload();
        if (!validateEvent(payload)) return;  // skip if duplicate

        ChatParticipant sender = findSender(payload);
        ChatMessage chatMessage = saveChatMessage(sender);
        saveReadStatuses(chatMessage, sender.getChatRoom());
    }

    private boolean validateEvent(ChatMessagePayload payload) {
        String chatKey = key + ":" + payload.getEventId();
        return StringUtils.hasText(redisTemplate.opsForValue().get(chatKey));
    }
}
```

Each event has a **unique ID (eventId)** generated by Redis. `validateEvent` checks if the key exists in Redis — already-processed events are skipped.

### Kafka Manual ACK

```java
@KafkaListener(topics = EventType.Topic.CHAT_MESSAGE,
    groupId = "#{T(java.util.UUID).randomUUID().toString()}")
public void chatMessageListen(String message, Acknowledgment ack) {
    Event<ChatMessagePayload> event = Event.fromJson(message);
    messageSendingOperations.convertAndSend(
        "/chatRoom/" + event.getPayload().getChatRoomId(), event.getPayload());
    ack.acknowledge();  // manual ACK
}
```

Explicitly calling `ack.acknowledge()` commits the offset only after processing completes. If a failure occurs during processing, Kafka redelivers the message, and Redis idempotency prevents duplicates.

## Result

- `chatRoomId` partition key ensures **per-room message ordering**
- Redis idempotency **prevents duplicate processing** in at-least-once delivery
- Manual ACK + Outbox retry achieves **0% message loss**

## Reflection

At-least-once guarantees messages are delivered "at least once" but not "exactly once." Redis idempotency bridges this gap.

Achieving **reliability (0% loss) + accuracy (0% duplicates) + ordering (per-room)** simultaneously required not a single technology but a combination of mechanisms working together.
